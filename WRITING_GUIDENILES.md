# Writing Guidelines

## Purpose

The purpose of this knowledge base is not to simplify technologies.

The goal is to help readers truly understand them.

Every document should help build an accurate mental model by explaining:

- Why a technology exists.
- How it works.
- Where it fits in real software systems.
- How it is used in production.

Understanding should always be the result of reasoning, never memorization.

---

# Core Principles

## 1. Never Assume Previous Knowledge

Never introduce an important concept without providing enough context for the reader to continue learning.

A reader should never stop because a concept appeared without explanation.

---

## 2. Build Knowledge Progressively

Knowledge should be built step by step.

Each section should naturally answer the questions generated by the previous one.

Learning should feel like a continuous conversation instead of a collection of independent topics.

---

## 10. Each Section Should Answer Only Its Own Questions

Every section has a single learning objective.

A section should answer only the questions that belong to it and avoid introducing conclusions that will be explored later.

Good learning progression creates curiosity.

Each section should naturally generate the question that the next section answers.

Example:

Definition

↓

"Why does Node.js exist?"

↓

How it Works

↓

"How can it efficiently handle thousands of concurrent operations?"

↓

How it Fits into the Ecosystem

↓

"Where does Node.js fit inside a real software system?"

↓

Real-World Usage

↓

"When should an engineering team choose Node.js?"

↓

Practical Examples

↓

"What does this look like in a production environment?"

By the end of the document, the reader should feel that every important question was answered at the right moment.

---

## 4. Explain Reasons Before Rules

Never tell the reader _what_ without first explaining _why_.

Readers should naturally reach conclusions through understanding.

The document should never ask the reader to memorize rules.

---

## 5. Teach Systems, Not Isolated Technologies

Technologies should never be explained as isolated tools.

Always explain:

- where they fit;
- what surrounds them;
- how they interact with other technologies;
- why they exist inside a production architecture.

The objective is to understand complete software systems.

---

## 6. Explain Concepts Before Implementation

Implementation details should only appear after the reader understands the underlying concepts.

Syntax is useful.

Concepts are essential.

---

## 7. Reduce Complexity Without Hiding It

Do not oversimplify.

Break complex ideas into smaller understandable concepts while preserving technical accuracy.

The objective is understanding, not simplification.

---

## 8. Real-World Examples First

Whenever possible, examples should represent production systems.

Prefer examples such as:

- REST APIs
- Authentication systems
- Background Workers
- Distributed Systems
- Cloud Architectures
- Queues
- Caching
- Frontend Applications
- Microservices

Avoid examples whose only purpose is demonstrating syntax.

---

## 9. Every Paragraph Should Answer a Question

Avoid writing isolated information.

Each paragraph should answer a question the reader naturally has at that point.

Good documentation feels like a conversation.

---

## 10. The Overview Should Build Understanding

The purpose of an Overview is not to summarize a technology.

Its purpose is to provide enough understanding that readers naturally reach the most important conclusions by themselves.

If the Overview is written correctly, a separate "Key Takeaways" section should not be necessary.

---

# Concept Classification

Every time a new concept is introduced, classify it before deciding how much to explain.

The classification depends on the technology currently being documented, not on the concept itself.

The same concept may be fundamental in one document and only contextual in another.

---

## Core Concept

A concept that is essential to understanding the current technology.

It must always receive a concise explanation before being used.

The reader should fully understand why the concept matters in the context of the current document.

Examples in a Node.js Overview:

- JavaScript
- Runtime
- V8
- Thread
- Single-threaded
- Event Loop
- Non-blocking I/O
- REST API

---

## Context Concept

A concept that helps explain the ecosystem but is not required to understand the technology itself.

Provide only enough information for the reader to understand why it appears.

Examples:

- AWS Lambda
- CLI
- Kubernetes
- Cron Jobs

---

## Knowledge Base Concept

A concept that already has, or will eventually have, its own dedicated document.

Do not explain it in depth.

Provide only the minimum context required for the reader to continue learning.

For now, highlight these concepts using **bold**.

Once dedicated documentation exists, replace the highlight with an internal link.

Example:

Node.js applications commonly communicate over **HTTP**, the protocol used by clients and servers to exchange requests and responses.

The purpose is to acknowledge the concept without interrupting the current learning flow.

---

# Document Types

The knowledge base is divided into two complementary document types.

Each one has a different purpose and depth.

---

# Overview

## Purpose

The purpose of an Overview is to help readers **understand the technology**.

After reading an Overview, readers should understand:

- Why the technology exists.
- The problem it was created to solve.
- How it works at a high level.
- Where it fits inside modern software systems.
- When it should (and should not) be used.
- How it is commonly applied in production.

The Overview should build an accurate mental model without exploring internal implementation details.

Readers should finish the document understanding the technology well enough to discuss it confidently and recognize it inside real software systems.

Typical reading time: **10–15 minutes**.

---

## Overview Structure

Every Overview follows the same structure.

1. Definition
2. How it Works
3. How it Fits into the Ecosystem
4. Real-World Usage
5. Practical Examples

Each section answers only its own learning objective and naturally prepares the reader for the next one.

---

# Deep Dive

## Purpose

The purpose of a Deep Dive is to help readers **master the technology from an engineering perspective**.

It does **not** teach how to build applications with the technology.

Instead, it explains the internal concepts that allow engineers to reason about the technology, discuss it confidently during technical interviews, and make informed architectural decisions.

A Deep Dive should answer the technical questions expected from a senior engineer.

Examples include:

- Internal architecture
- Execution model
- Performance
- Memory management
- Concurrency
- Synchronization
- Internal components
- Design decisions
- Common misconceptions
- Real-world trade-offs

Each concept should be developed independently, following the same writing principles as the Overview:

- Introduce one concept at a time.
- Define every new concept before using it.
- Avoid unnecessary transitions.
- Maintain high knowledge density.
- Build understanding progressively.

The goal is not to memorize implementation details.

The goal is to understand how the technology works well enough to reason about it.

Typical reading time: **30–60 minutes**, distributed across independent concepts.

---

# Relationship Between Both Documents

The Overview explains what every software engineer should know.

The Deep Dive explains what every senior software engineer should understand.

The Overview builds the mental model.

The Deep Dive expands and strengthens it.

The purpose of a Deep Dive is to explain how a technology actually works internally.

Topics may vary depending on the technology, but generally include:

- Internal Architecture
- Internal Components
- Execution Flow
- Performance
- Security
- Best Practices
- Common Mistakes
- Real-World Examples
- References

The Deep Dive should never repeat the Overview.

It should expand it.

---

# Final Goal

Every document should leave the reader with fewer unanswered questions than when they started.

The objective of this knowledge base is to build engineers who understand technology deeply enough to reason about it, not simply remember it.
