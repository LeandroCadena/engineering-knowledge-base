# Writing Guidelines

## Purpose

The purpose of this knowledge base is not to simplify technologies.

The goal is to truly understand them.

Every document should help the reader build an accurate mental model of a technology by explaining **why it exists, how it works, where it fits in real software systems, and how it is applied in production**.

Understanding should always be the result of reasoning, never memorization.

---

# Core Principles

## 1. Never Assume Previous Knowledge

Never introduce an important concept without explaining it first.

If a concept is essential to understanding the current topic, it must either:

- be briefly explained;
- or reference another document in the knowledge base (when available).

Readers should never get stuck because of hidden assumptions.

---

## 2. Build Knowledge Progressively

Knowledge should be built step by step.

Each section should naturally prepare the reader for the next one.

Every new concept should feel like the natural answer to a question generated by the previous section.

Learning should feel like a continuous conversation.

---

## 3. Explain Reasons Before Rules

Never tell the reader _what_ without first explaining _why_.

Example:

❌ Node.js should be used for I/O-bound applications.

✅ Node.js uses an event-driven, single-threaded execution model designed to spend most of its time waiting for I/O operations. Because of this architecture, it performs particularly well for I/O-bound workloads.

Readers should arrive at conclusions through understanding, not authority.

---

## 4. Teach Technologies as Parts of Systems

Technologies should never be explained as isolated tools.

Always explain:

- where they fit;
- what surrounds them;
- how they interact with other technologies;
- why they exist inside a production architecture.

The goal is to understand complete software systems, not individual technologies.

---

## 5. Explain Concepts Before Implementation

Implementation details should only appear after the reader understands the underlying concepts.

Syntax is important.

Concepts are essential.

---

## 6. Reduce Complexity Without Hiding It

Do not oversimplify.

Instead, break complex ideas into smaller concepts while preserving technical accuracy.

The objective is understanding, not simplification.

---

## 7. Real-World Examples First

Whenever possible, examples should represent real production systems.

Prefer examples such as:

- REST APIs
- Authentication systems
- Distributed systems
- Cloud architectures
- Workers
- Queues
- Caching
- Frontend applications
- Microservices

Avoid examples that only demonstrate syntax without context.

---

## 8. Every Paragraph Should Answer a Question

Avoid writing isolated information.

Each paragraph should answer a question the reader naturally has at that point.

Good documentation feels like a conversation, not a textbook.

---

## 9. The Overview Should Build Understanding

The purpose of an Overview is not to summarize a technology.

It is to provide enough understanding that the reader can naturally reach the most important conclusions without needing a dedicated "Key Takeaways" section.

If a reader understands the Overview, they should already be able to explain the technology in their own words.

---

# Overview Structure

Every Overview should follow this structure:

1. Definition
2. How it Works
3. How it Fits into the Ecosystem
4. Real-World Usage
5. Practical Examples

Each section exists because the previous one naturally creates the need for the next.

---

# Deep Dive Structure

The purpose of a Deep Dive is to explain how a technology actually works internally.

Topics may vary depending on the technology, but generally include:

- Internal Architecture
- Internal Components
- Execution Flow
- Performance
- Security
- Best Practices
- Common Mistakes
- Real-World Examples
- References

The Deep Dive should never repeat the Overview.

It should expand it.

---

# Final Goal

Every document should leave the reader with fewer unanswered questions than when they started.

The ultimate objective of this knowledge base is to build engineers who understand technology deeply enough to reason about it, not simply remember it.
